# Import Diff Contract

Import is the local review surface for parsed timetable changes, unresolved/fallback confirmations, and user overrides. It is intentionally separate from Home provider apply: Import adopts local preview state, while Home performs Google Calendar writes.

## 1. Inputs

Import consumes the current workspace preview result:

- parsed source catalog state;
- selected class;
- effective first-week mapping;
- effective time-profile resolution;
- parser warnings, diagnostics, and unresolved items;
- normalized occurrences and export groups;
- local snapshot diff;
- provider preview context when available;
- saved course schedule and presentation overrides.

The page must refresh when any source file slot is replaced, removed, or updated by drag/drop.

## 2. Grouping model

Import projects raw changes into a grouped review model:

```text
course group
  repeat-rule group
    occurrence rows
```

Grouping is presentation-only. Raw `Added`, `Updated`, and `Deleted` change IDs remain authoritative for selection, local adoption, and provider apply handoff.

Course groups may contain selectable change rules and unchanged context rules at the same time. Parent checked/partial state must be computed only from selectable changes.

## 3. Pinned review items

These items appear above ordinary change groups:

- same-campus time-profile fallback confirmations;
- local schedule conflicts generated by user overrides;
- unresolved regular course blocks that require explicit confirmation.

Pinned items remain visible even when ordinary filters would otherwise hide them. Confirming an unresolved item through the inline editor removes it from the pinned area and regenerates the preview so it can appear as a valid occurrence.

## 4. Change selection

- Ready changes default to selected.
- A change blocked by a fallback or unresolved confirmation remains unchecked until confirmed.
- `Select current page` toggles visible selectable rows in the current filter result.
- Course and repeat-rule selectors affect selectable child rows only.
- Clicking an indeterminate selector should resolve to a clear selected/unselected state rather than leaving it stuck.
- Unmanaged Google Calendar context rows are never selectable provider changes.

## 5. Detail panel scope

The detail panel supports five selection scopes:

1. course group;
2. repeat-rule group;
3. single occurrence;
4. inline course settings;
5. pinned fallback/unresolved confirmation.

Course-group details are built from the currently parsed timetable occurrences, not from provider state. They list parsed repeat rules and preserve sparse week-expression gaps; for example, `3-9周,11-20周` must not be collapsed into a single continuous recurrence.

Repeat-rule details list the concrete occurrences represented by that rule. Selectable child changes keep their checkboxes; unchanged context rows remain read-only context until the user selects a specific occurrence for detail.

Single-occurrence details apply edits only to that occurrence unless the user changes the editor to a recurring rule. In that case, the save becomes a rule-level override and clears the single-date binding.

Course settings expose provider presentation choices such as time zone and Google Calendar color, followed by parsed repeat-rule context. They should not show diff-only before/after chrome that belongs to single-occurrence details.

## 6. Editing and overrides

Import edits are local. They update workspace preferences/overrides and rebuild the local preview.

Required behavior:

- preserve occurrence wall-clock date/time when saving time-zone changes;
- use the configured default Google Calendar IANA time zone when a parsed occurrence has no explicit time-zone override;
- allow every-N day/week/month/year recurrence edits, weekly weekday selection, and monthly day-of-month or last-weekday patterns;
- treat biweekly as every 2 weeks;
- auto-swap inverted start/end dates on save;
- show one date for one-time rules;
- keep inline text fields compact until content wraps;
- allow delete details to cancel a delete by writing a one-time local override;
- merge generated override occurrences that duplicate an existing same-course, same-local-time, same-location slot;
- pin conflicts when different courses share the same local date/time.

Preview rebuilds triggered by save/reset should be latest-operation-wins so an older in-flight refresh cannot overwrite the newest state.

## 7. Notes and metadata rendering

Google Calendar description diffs should be shown as a single-column code-style line diff for updates. Added and deleted rows do not need a note diff block because only one side exists.

Editable note payload lines may be edited inline when they represent user notes. App-managed metadata lines such as `managedBy`, `localSyncId`, `sourceFingerprint`, and `sourceKind` are read-only display/review text.

Remote ownership and destructive sync identity must never be inferred from ordinary description text. The provider layer trusts private extended properties and local mappings.

Parsed note fragments should be normalized for display:

- explicit `Notes:` / `备注:` fields become readable note text;
- continuation lines remain part of the same note;
- slash-delimited tail segments such as class size, assessment mode, course-hour composition, or credits remain visible instead of becoming `No notes`;
- legacy short fragments may be expanded into labeled parts before comparison.

If an update changes only the description payload, summaries should show a note update rather than inventing field changes for metadata already embedded in the notes.

## 8. Unchanged schedules

`Unchanged Schedules` remains the trailing local context area.

When ordinary changes exist:

- unchanged courses with no planned changes appear after the change list;
- unchanged rules for a changed course stay inside that course group as context;
- the same course should not be duplicated in both the change group and the trailing unchanged area.

The unchanged area supports mutually exclusive repeat-rule and all-times modes. At least one mode must remain selected. Selecting a concrete occurrence from a repeat-rule detail should switch the left side to all-times mode, scroll to the occurrence in the shared review pane, and highlight it.

## 9. Layout and localization

Import should support compact, medium, and expanded layouts without hiding required actions. The grouped review list and unchanged schedules share one left-side scroll flow so long pages remain usable.

User-visible Import strings must come from localization resources. Filter/group/sort behavior must use stable semantic values rather than localized text. Chinese may appear only as exact UI/domain text or CQEPC source examples.

## 10. Reset behavior

The top reset/default action may clear:

- saved course schedule overrides;
- per-course presentation overrides such as time zone or color;
- optional provider task-generation rules back to disabled defaults;
- resettable non-added local snapshot drift.

It must not silently hide real `Added` rows from the current parsed timetable.
